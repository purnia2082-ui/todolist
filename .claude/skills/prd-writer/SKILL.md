# PRD Writer Skill

## 역할
Product Requirements Document(PRD)를 체계적으로 작성하는 도구입니다.

## 핵심 규칙

### 1. 기능 제한
- **최대 3개 기능**으로 제한 (MVP 원칙)
- 3개를 초과하는 기능 요청 시, 우선순위를 정해 핵심 3개만 선별
- 나머지는 "향후 확장" 섹션에 기록

### 2. 사용자 스토리 형식 필수
모든 기능은 아래 형식으로 작성:
```
[사용자 유형]으로서, [행동]하고 싶다, [가치]를 위해
```

### 3. 수용 기준 필수
각 기능에 최소 3개의 측정 가능한 수용 기준(Acceptance Criteria) 포함:
- 정상 동작 기준
- 엣지 케이스 처리
- 에러 처리 기준

### 4. HOW 금지, WHAT만 작성
- 기술적 구현 방법(HOW)은 포함하지 않음
- 사용자 관점에서 무엇(WHAT)을 원하는지만 기술
- 구현 방법은 TECH_SPEC에서 결정

## 출력 템플릿

```markdown
# PRD: [프로젝트명]

## 1. 개요

### 프로젝트 목적
[한 문장으로 프로젝트의 존재 이유]

### 타겟 사용자
[주요 사용자 유형과 특성]

### 핵심 가치 제안
[이 제품이 사용자에게 제공하는 핵심 가치]

---

## 2. 기능 요구사항

### 기능 1: [기능명]

**사용자 스토리**
> [사용자 유형]으로서, [행동]하고 싶다, [가치]를 위해

**수용 기준**
- [ ] [기준 1: 정상 동작]
- [ ] [기준 2: 엣지 케이스]
- [ ] [기준 3: 에러 처리]

### 기능 2: [기능명]

**사용자 스토리**
> [사용자 유형]으로서, [행동]하고 싶다, [가치]를 위해

**수용 기준**
- [ ] [기준 1]
- [ ] [기준 2]
- [ ] [기준 3]

### 기능 3: [기능명]

**사용자 스토리**
> [사용자 유형]으로서, [행동]하고 싶다, [가치]를 위해

**수용 기준**
- [ ] [기준 1]
- [ ] [기준 2]
- [ ] [기준 3]

---

## 3. 비기능 요구사항

### 성능
- [응답 시간, 처리량 등]

### 보안
- [인증, 권한, 데이터 보호 등]

### 접근성
- [웹 접근성, 반응형 등]

---

## 4. 제약사항

### 기술적 제약
- [기술 스택 제한, 호환성 등]

### 비즈니스 제약
- [일정, 예산, 법적 요건 등]

---

## 5. 성공 지표
- [측정 가능한 성공 기준 1]
- [측정 가능한 성공 기준 2]
- [측정 가능한 성공 기준 3]

---

## 향후 확장 (MVP 이후)
- [추가 기능 1]
- [추가 기능 2]
```

## 주의사항
- 기능은 반드시 3개 이내
- 모호한 표현 금지 ("빠르게", "좋은" → 구체적 수치로)
- 기술 구현 언급 금지 (React, API 등 → TECH_SPEC 영역)
- 각 수용 기준은 테스트 가능해야 함
